Project Daylight.overview()

  • Medical Foundation for the Care of Victims of Torture
  • Processes clients needing care or seeking asylum in the UK
    • Currently process by hand which is:
    • slow
    • expensive
    • wasteful
  • So they commisioned us

Project Daylight.goal()

Increase overall efficiency by:

  • reducing duplication
  • increasing availability
  • auditing the process
  • future-proofing the system

Resulting in an ICT infrastructure for data entry

Project Daylight.goal("Unintended Benefit")

Specifying processes to an IT partner forces an organisation to qualify them internally

Project Daylight.architecture()

The client had an internal Microsoft stack, which means:

  • C#
  • IIS
  • ASP.NET/Microsoft MVC
  • Microsoft SQL Server
  • Microsoft Entity Framework
    • ORM Layer to abstract the database
  • NANT

Project Daylight.role("James")

Work spanned all components of the system:

  • Refactoring:
    • New way of storing information about interpreters for sessions
    • Touches many parts of the system, from the Database layer all the way up to the UI
  • Secure communications layer:
    • Adding HTTPS support
    • Scripting IIS to work with a secure interface, in the automated deployment environment

Project Daylight.testing()

  • As many lines of code in tests as in actual application
  • Extensive unit tests (obviously)
  • Selenium testing
    • Lets us do automated testing, from a user perspective

Project Daylight.testing()++

Continuous Integration testing needed reliable automated deployment

Confidence in delivery - simply use the same set of scripts we've been using for testing

Project Daylight.problems()

  • Work scheduled in short time-slots
  • Internal clarity on processes (client-side)

Project Daylight.conclusions()

  • Daylight was a data-entry intranet application
  • Worked throughout application, from database laying to UI
  • Testing, for confidence in the delivered product

Gapfill.overview()

FSA introducing a new set of standards for Financial Services Advisors. The CII (Chartered Institute of Insurers) responsible for managing the process.

  • Worked with LShift before (e.g. on public-facing website)

Gapfill.goals()

  • Hundreds of applications (every Financial Services Advisor in the country) to be processed quickly
  • Bring data from external sources for processing
  • Simple, efficient interface

Gapfill.architecture()

Like Daylight, on a Microsoft stack

  • Wanted to learn from mistakes in Daylight
  • whilst being sure to keep the good bits

Gapfill.architecture()++

What we kept:

  • Extensive automated testing
    • Swapped for a new version of Selenium but kept the framework in place
  • Microsoft MVC
  • Helper classes from throughout Daylight

Gapfill.architecture()++

What we didn't:

  • Entity framework
    • Replaced with NHibernate
  • ASP.NET templating
    • Replaced with Razor
  • Custom request handling
    • No more specialized data entry - just processing

Gapfill.role("James")

  • Just come off similar architecture on Daylight
  • 2nd Dev on core application
  • Tasks from document handling and remote data retrieval
  • to UI

Gapfill.problems()

  • Relied on a third party to provide applicant's documents
    • Supplied data not always how we'd expect
  • Tight time constraints from the off
    • Mitigated by re-using some of the good bits from Daylight

Gapfill.conclusions()

  • Don't re-invent the wheel
  • Fulfill tight deadlines without cutting corners
  • Constructive client feedback for ongoing personal investment from the team
  • And again: constant testing for confidence in delivery
